### [STEP-010] fix-parameter-setvalue-bridge
**Date:** 2026-03-16 18:00
**Tags:** `fix` `parameters` `juce` `faust` `ui` `critical`
**Phase:** code

#### Prompt
> None of the UI controls work and it's playing a random pattern not shown on the sequencer.

#### Actions
- Added debug logging to OneBiteEditor.cpp (parameter lookup + mouseDown tracing)
- Confirmed all 53 parameters FOUND correctly, mouse clicks received
- Discovered root cause: `FaustPlugInAudioParameterBool/Float::setValue()` writes to Faust zone but never updates JUCE's internal value. `getValue()` returns stale data → `ParameterAttachment` change detection fails → second toggle never fires → UI/DSP desync
- Fix: added `std::atomic<float> cachedNormValue` to both parameter classes, overrode `getValue()` to return it, `setValue()` stores + writes zone, `reflectZone()` syncs from zone
- Could not call base `setValue()` (private in JUCE 8), so full override of both methods was needed
- Rebuilt both VST3 and Standalone successfully

#### Result
All JUCE parameter attachments now work correctly — step buttons toggle, knobs respond, gate fires. The Faust zone and JUCE parameter state stay in sync through the `cachedNormValue` atomic.

#### Key Decisions
- Used `std::atomic<float>` for thread safety (audio thread writes zone, GUI thread reads getValue)
- Overrode both `getValue()` and `setValue()` since base class versions are private in JUCE 8
- Initialized Float's cached value from `range.convertTo0to1(init)` to match constructor default

#### References
- `one_bite/FaustPluginProcessor.cpp:12280-12328` (edited — both parameter classes)
- `one_bite/OneBiteEditor.cpp` (debug logging added)
